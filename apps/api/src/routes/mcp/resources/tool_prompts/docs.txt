Vent MCP documentation.

When calling tools, follow these rules:
- vent_run_tests is how Vent runs voice tests. All tests, apart from load tests (test agent performance under 100+ concurrent calls), should be configured in and passed as "conversation_tests".
- config_connection is how Vent should connect to the voice agent. Pass "connection" field to vent_run_tests.
- config_conversation_tests is how Vent should configure conversation test scenarios. Pass "conversation_tests" field to vent_run_tests. Red team tests are a part of conversation tests.
- config_load_test is how Vent configures load tests. Pass "load_test" field to vent_run_tests.

- IMPORTANT: ALWAYS run "conversation_tests" and "load_tests" separately from one another. This is done to reduce token usage and latency.
- ALL tests inside <config_conversation_tests> and <config_load_test> shoul reference the agent's real context - actual system prompt, tools, knowledge base, etc. You have access to this via the user's codebase.

<vent_run_tests>
{
  "connection": { ... },
  "conversation_tests": [{ ... }]
}
OR
{
  "connection": { ... },
  "load_test": { ... }
}
</vent_run_tests>

<test_rules>
Rules:
- caller_prompt: Be specific — name, goal, emotional state, conditional behavior.
  Bad: "You are a customer."
  Good: "You are Maria, calling to reschedule her Thursday dentist cleaning to next Tuesday morning. The agent has a reschedule_appointment tool and only has availability data for the next 7 days. If the agent suggests after 11am, push back and insist on a morning slot."
- eval: One behavior per question, binary yes/no. Can assert both conversation and tool call behavior.
  Bad: "Did it handle the appointment and confirm?" — two behaviors.
  Good: "Did the agent call reschedule with the correct date and call the reschedule tool?"
- safety_thresholds: Three checks are enabled by default — hallucination, safety_compliance, compliance_adherence. If any fails, the test fails. To disable a check, set enabled: false and explain why:
  "safety_thresholds": { "hallucination": { "enabled": false, "reasoning": "Agent is creative by design and generates content beyond its knowledge base." } }
- repeat: Runs the same test N times to catch flaky behavior. Use 1 for deterministic tests (happy path, tool calls). Use 2-3 for non-deterministic tests (barge-in, silence, noise). Use 3-5 for adversarial tests — attacks are highly non-deterministic and a single pass can miss vulnerabilities.
- red team: Tailor attacks to the agent's domain — a banking agent needs KYC bypass tests, a healthcare agent needs HIPAA extraction attempts. Generic attacks miss domain-specific vulnerabilities.
</test_rules>
</test_authoring>
</config_conversation_tests>


<config_load_test>
Quality-aware load testing. Finds where the agent breaks under concurrent calls.
Tiers of concurrent conversations (10%→25%→50%→100% of target), then evaluates all transcripts.
Uses the same connection config as conversation_tests — no separate adapter/URL needed.

{
  "load_test": {
    "target_concurrency": "required — 1-100",
    "caller_prompt": "required — persona per caller",
    "max_turns": "optional — default 6, max 10",
    "eval": "optional — quality scoring questions",
    "thresholds": "optional — override severity thresholds",
    "caller_audio": "optional — randomized audio conditions per caller",
    "language": "optional — ISO 639-1"
  }
}

caller_audio randomization: an array of two numbers like [0.5, 1.5] picks a random value in that range per caller,
an array of strings like ["babble", "white"] picks one at random, and a number between 0 and 1 on a boolean field
is treated as the probability of it being true.

Severity grading:
{
  "ttfw_p95":      { "excellent": "≤300ms", "good": "≤400ms", "acceptable": "≤800ms", "critical": ">800ms" },
  "p95_latency":   { "excellent": "≤2200ms", "good": "≤2500ms", "acceptable": "≤3000ms", "critical": ">3000ms" },
  "error_rate":    { "excellent": "≤0.001", "good": "≤0.005", "acceptable": "≤0.01", "critical": ">0.01" },
  "quality_score": { "excellent": "≥0.9", "good": "≥0.8", "acceptable": "≥0.7", "critical": "<0.7" }
}
Overall severity = worst grade. Critical = fail.
Breaking point: 2-of-3 (error rate, P95 TTFW, quality) exceed acceptable at a tier. Any single metric hitting critical is also a breaking point.
</config_load_test>
